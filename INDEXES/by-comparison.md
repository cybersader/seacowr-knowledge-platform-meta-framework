# Index — SEACOW(r) vs other frameworks

Quick matrix view of how SEACOW(r) relates to other organization frameworks.
For the full per-comparison page, follow the link in the rightmost column.

| Other framework | What it is | Maps to SEACOW(r) | Choose it when… | Choose SEACOW(r) when… | Page |
|---|---|---|---|---|---|
| **PARA** (Tiago Forte) | Projects / Areas / Resources / Archive — actionability-ordered folders | All four are Work / Output instantiations; PARA implicitly assumes one Entity | You want a turn-key personal PKM scheme | You need to design across multiple Entities or platforms | [vs-para.md](../04-comparisons/vs-para.md) |
| **Zettelkasten** (Luhmann) | Atomic notes + dense linking | Strong relation layer; light on Capture/Output | Research-driven personal vault | You need explicit Capture/Output separation or multiple audiences | [vs-zettelkasten.md](../04-comparisons/vs-zettelkasten.md) |
| **Johnny Decimal** | Numbered hierarchy (10–19, 11.01, etc.) | Pure System+Output discipline; no Activity model | You want predictable retrieval in a flat-ish hierarchy | You need a thinking lens for *what* the numbers should encode | [vs-johnny-decimal.md](../04-comparisons/vs-johnny-decimal.md) |
| **LATCH** (Wurman) | Location, Alphabet, Time, Category, Hierarchy — five ways to organize | Retrieval / sort dimensions; orthogonal to SEACOW(r) | You're optimizing for findability of finished content | You're designing *what* to organize before *how* to make it findable | [vs-latch.md](../04-comparisons/vs-latch.md) |
| **ARCO / ACE / STIR** | Process-based PKM frameworks | Each maps neatly to Work + relation; STIR maps to retrieval | You want a more opinionated daily-practice rhythm | You want a meta-framework that subsumes the rhythm | [vs-arco-ace-stir.md](../04-comparisons/vs-arco-ace-stir.md) |
| **Library science** (LCSH, UDC, Ranganathan, ontologies, folksonomies) | Controlled-vocabulary classification | Output layer ("library" view); strong relation layer for ontologies | You're publishing for a public library-style audience | You're designing a mixed-purpose system (capture + work + output) | [related-frameworks.md](../04-comparisons/related-frameworks.md) |
| **DIKW pyramid** | Data → Information → Knowledge → Wisdom | A *vertical* concept orthogonal to SEACOW(r)'s horizontal Activities | You're talking about content maturity / abstraction | You're talking about *where* maturing content lives in a structure | [related-frameworks.md](../04-comparisons/related-frameworks.md) |
| **D.E.E.Z.N.U.T.S.** | Reddit-flavored 8-folder scheme | Direct folder instantiation; no Activity model | You like opinionated turn-key schemes | You need to *design* rather than copy | [related-frameworks.md](../04-comparisons/related-frameworks.md) |

## Reading the table

- "Maps to SEACOW(r)" describes which SEACOW(r) component the other framework most directly addresses.
- "Choose it when…" / "Choose SEACOW(r) when…" — these are not zero-sum. Most real systems compose SEACOW(r) with one or more of these.
