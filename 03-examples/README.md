## What makes a good example

- It names the SEACOW(r) component each structural choice serves.
- It explains *why* — what the audience or platform constraint demanded.
- It's honest about the parts that don't fit the framework (rather than forcing them).

## Anti-pattern

Don't post a folder tree without rationale. A tree alone is decoration. The
value is in the *mapping* from SEACOW(r) component → structural decision →
trade-off accepted.
