# Vault folder & tag structure — working notes

Earlier working notes on how to structure folders *and* tags together in
cyberbase. These are looser-grade brainstorm material that pre-dates the
formal [tag-structure.md](tag-structure.md) and [folder-structure.md](folder-structure.md).
They're kept here for the design rationale.

## The workflow

**ARC** — Add, Relate, Communicate. (Nick Milo's PKM rhythm.)

# Ideas

## Folders

### Good ways to structure folders

- Most to least actionable (e.g. PARA).
- Meta folders before or after (e.g. obsidian utils).
- MOC files to link or map a tag-based ontology.
- Separate by the part of the workflow (ARC) — e.g. Inbox, MOCs, Hierarchies / Docs.
- Don't organize time manually and granularly. Have an archive under time where you can throw those. That means *don't duplicate time-based under some archive-focused folder*.
- Do you duplicate ontologies under archives or no?
- **Use folders to structure the knowledge work process and not the content.**
- Structure for recall.

### High-level structuring priorities

- Make sense for the audience.
- Order of high level should mean something.
  - Based off of LATCH:
    - **L**ocation, **A**lphabet, **T**ime, **C**ategory (cluster), **H**ierarchy (ontology).

### Root-level folder ideas

- PARA.
- ACE.
- Efforts, Journal/Time, Knowledge (EJK).

### Folder notes or no?

- Without enforcing them you have images above the note files.
- Enforcing them is hard, though.

### Outputs, docs, audience

- If you have to, leave shortcuts to the consumable and deliverable parts.
- Maps of content.
- Docs / output areas sorted for different audiences.
- MOCs for different audiences.
- There's always the search bar.

> Start simple, then scale up or down as needed. Seriously.

## Tags

Just as folders need structures, so do tags — but they can overlap.

### Good ways to structure tags

- By the content.
- With unique names.
- Ontology-based.
- For clustering.
- For filtering with MOCs.
- By the process.
- By actionability.
- For recall — LATCH components.

### Ideas for tag structures

- Use underscore to auto-complete into ontology-focused or process-focused tag structures.
- Use meta tag categories.
- Use a tag root for meta stuff for working in the vault.

### What's supported for tags?

See the [Obsidian help — Tags](https://help.obsidian.md/tags) reference and the cyberbase notes on [Obsidian tag limitations](../../02-tools/platform-notes/obsidian-tag-limitations.md).

### Tag case

- Just use snake-case since double-clicking and programming with the tag names becomes easier.

### Tag structure examples

- `_/unstructured categories or tags`
  - Alphabetical or autocomplete sorting can also help here.
- `(number or emoji)_word_here/ontologies`
  - Or structured strictly where the alphabetical or chronological naming helps.
- `words/stuff` — based on information process or KMS process or functionality (e.g. clippings).
  - These have to be easy to remember.
- `-/other_purpose`
- `--/yet_another_section`

If you need to use regular words for unstructured tags then you can hide some things in the underscore or dash tags.

The first type of character helps with autocomplete.
